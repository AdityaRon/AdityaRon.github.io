Design → Outcome

How I work — design first, then ship it.

Two efforts where I wrote the design up front and drove it to production. The point isn't just the result — it's that the architecture I proposed is the one that shipped.

Re-platforming a security correlation engine

From a warehouse-coupled monolith to an open lakehouse — by design

A correlation engine I help found and architect runs 400+ detections an hour — resolving entities and correlating them in an ephemeral Neo4j graph into high-fidelity incidents. It was coupled to a proprietary data warehouse — costly, and hard to scale the ingest and analysis stages independently. I wrote the migration design before we touched code.

◆ The design I authored

  • Split the monolith into a producer (pre-aggregates detection "edges" into an open Iceberg lakehouse) and a consumer (graph analysis → alerts) — so each scales independently.
  • Single-scan grouped caching: read each source once, derive every policy off the cached base, union, write once — collapsing 155 repeated scans of one table into one.
  • Per-pipeline triggering, managed reference-data tables, and metrics parity carried over from the legacy system.

▲ What shipped

  • The producer/consumer split landed in production, validated end-to-end on real tenants.
  • The single-scan strategy shipped as a one-pass query rewrite across 92 detection policies, byte-for-byte identical output.
  • Reference data moved to managed lakehouse tables; observability and alerting reached parity.
